## 摘要
智能体（Agents）是由大语言模型（LLM）驱动的组件，能够以强大且任意的方式改变环境。由于异步和故障，在生产环境中为智能体的执行提供保障具有挑战性。在本文中，我们提出了一种名为 LogAct 的新抽象，其中每个智能体都是一个播放共享日志的解构状态机。在 LogAct 中，智能体动作在执行前在共享日志中可见；可以通过可插拔、解耦的投票器在执行前拦截；并在智能体或环境发生故障时实现一致性恢复。LogAct 实现了智能体内省，允许智能体利用 LLM 推理分析自身的执行历史，从而实现恢复、健康检查和优化的语义变体。在评估中，LogAct 智能体能够高效且正确地从故障中恢复；调试自身性能；优化集群中的 Token 使用；并在代表性基准测试中，仅以 3% 的良性效用下降为代价，拦截了目标模型的所有非预期动作。

## Abstract
Agents are LLM-driven components that can mutate environments in powerful, arbitrary ways. Extracting guarantees for the execution of agents in production environments can be challenging due to asynchrony and failures. In this paper, we propose a new abstraction called LogAct, where each agent is a deconstructed state machine playing a shared log. In LogAct, agentic actions are visible in the shared log before they are executed; can be stopped prior to execution by pluggable, decoupled voters; and recovered consistently in the case of agent or environment failure. LogAct enables agentic introspection, allowing the agent to analyze its own execution history using LLM inference, which in turn enables semantic variants of recovery, health check, and optimization. In our evaluation, LogAct agents recover efficiently and correctly from failures; debug their own performance; optimize token usage in swarms; and stop all unwanted actions for a target model on a representative benchmark with just a 3% drop in benign utility.

*   **研究背景**：LLM 智能体（Agents）通常遵循“观察-推理-动作”的循环。然而，在生产环境中，智能体面临三大挑战：**安全性**（可能执行删除数据库等破坏性动作）、**故障恢复**（执行中途崩溃导致系统处于中间状态）和**可审计性**（难以追踪动作背后的逻辑）。

### 2. 方法论：LogAct 架构
*   **核心思想**：将智能体抽象为一个**在共享日志（AgentBus）上运行的解构状态机**。智能体不再是直接执行动作的程序，而是将“意图”写入日志，由多个解构的组件共同维护状态。
*   **关键技术细节**：
    *   **AgentBus（共享日志）**：一个强类型的、追加式的持久化日志。所有推理输出、意图、投票、决策和执行结果都作为不同类型的条目存入日志。
    *   **解构状态机**：将智能体拆分为四个独立组件：
        1.  **Driver（驱动器）**：负责 LLM 推理并生成“意图”（Intention）。
        2.  **Voter（投票器）**：审查意图的安全性，支持基于规则、静态分析或 LLM 的插件化投票。
        3.  **Decider（决策器）**：根据投票策略（如法定人数、一票否决）决定是否提交（Commit）或中止（Abort）。
        4.  **Executor（执行器）**：仅在看到“提交”条目后，在沙箱中执行动作。
    *   **智能体内省（Agentic Introspection）**：智能体可以利用 LLM 对自身的执行历史日志进行实时推理。这支持了**语义恢复**（分析崩溃前的状态并生成补偿动作）、**语义健康检查**（诊断性能瓶颈）和**群体优化**。

### 6. 主要结论与发现
*   **安全性**：通过“规则+LLM”的双投票机制，LogAct 成功拦截了目标模型在 AgentDojo 中的所有恶意动作，且良性任务的成功率仅下降了 3%。
*   **故障恢复**：利用语义内省，智能体能识别崩溃位置并优化低效代码。在实验中，恢复后的智能体通过更换算法，使执行速度提升了 **290 倍**。
*   **集群效率**：引入内省监督者后，6 智能体集群的任务产出提升了 17%，同时 Token 消耗降低了 41%。
*   **系统开销**：日志记录的存储开销极低（约 2.6KB/s），延迟主要受 LLM 推理限制，日志系统本身的延迟几乎可忽略。

### 7. 优点与亮点
*   **架构创新**：首次将共享日志与智能体状态机结合，解决了智能体动作“不可撤回”和“不可见”的痛点。
*   **解耦设计**：将执行器（Executor）与其他组件物理隔离，即使执行器被提示注入控制，也无法修改投票策略或伪造决策。
*   **语义化能力**：超越了传统的二进制恢复，利用 LLM 的推理能力处理复杂的、非结构化的故障恢复逻辑。

### 8. 不足与局限
*   **推理依赖**：语义恢复和投票高度依赖 LLM 的推理质量，若模型本身发生严重幻觉，恢复逻辑可能失效。
*   **延迟增加**：虽然日志系统开销小，但引入额外的投票步骤（尤其是 LLM 投票）会显著增加端到端延迟（实验显示双投票器增加了约 82% 的延迟）。
*   **并发控制**：目前主要侧重于单智能体可靠性，对于多个智能体同时修改同一外部环境时的复杂并发冲突（如分布式锁）讨论较少。
*   **应用限制**：对于实时性要求极高（毫秒级）的场景，多级投票机制可能过于沉重。
